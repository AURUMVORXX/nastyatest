### Реляционные базы данных (SQL)
Данные хранятся в таблицах с строгими связями между ними. Представьте себе Excel-таблицы, где каждая таблица связана с другими.

**Основные черты**:
- Таблицы - как листы в Excel
- Строки - отдельные записи
- Столбцы - поля данных
- Строгая структура - заранее определенные типы данных
- Связи между таблицами

**Примеры**: MySQL, PostgreSQL, SQLite

**Когда использовать**:
- Интернет-магазины (заказы, пользователи, товары)
- Банковские системы
- Любые данные со сложными связями

### Нереляционные базы данных (NoSQL)
Гибкие хранилища данных без жесткой структуры таблиц.

**Типы NoSQL**:
- Документные - хранят JSON-подобные документы
- Ключ-значение - как словарь в программировании
- Колоночные - оптимизированы для аналитики
- Графовые - хранят связи между объектами

**Примеры**: MongoDB, Redis, Cassandra

**Когда использовать**:
- Социальные сети
- Кэширование
- Big Data
- Быстро меняющиеся 

### Primary Key
Уникальный идентификатор каждой записи в таблице. Как номер паспорта для человека.

**Особенности**:
- Уникальный для каждой записи
- Не может быть NULL
- Одна таблица = один primary key

**Пример**:
- ``user_id`` в таблице пользователей
- ``order_id`` в таблице заказов

### Foreign Key
Ссылка на primary key другой таблицы. Создает связь между таблицами.

**Пример**:
В таблице заказы поле ``user_id`` - это foreign key, который ссылается на ``user_id`` в таблице пользователи.

Зачем нужно:
- Поддерживает целостность данных
- Не даст удалить пользователя, если у него есть заказы
- Автоматически обновляет связанные данные

### INNER JOIN
Показывает только те записи, которые есть в ОБЕИХ таблицах.

**Аналогия**: Представьте две группы людей - "Студенты" и "Те, кто сдал экзамен". INNER JOIN покажет только студентов, которые сдали экзамен.

### LEFT JOIN
Показывает ВСЕ записи из левой таблицы и совпадающие из правой.

**Аналогия**: Все студенты (левая таблица) и их оценки (правая таблица). Если студент не сдавал экзамен, его имя все равно покажется, но оценка будет NULL.

### RIGHT JOIN
Противоположность LEFT JOIN - все записи из правой таблицы и совпадающие из левой.

На практике используется реже, обычно вместо него используют LEFT JOIN, поменяв таблицы местами.

### Индексы
Как оглавление в книге или индекс в конце учебника. Помогает быстро находить данные без полного перебора.

Как работает:
- Без индекса: БД просматривает ВСЕ записи (как читать всю книгу)
- С индексом: БД сразу переходит к нужной странице

Типы индексов:
- **Простой индекс** - на одно поле
- **Составной индекс** - на несколько полей
- **Уникальный индекс** - гарантирует уникальность
- **Полнотекстовый индекс** - для поиска по тексту

**Плюсы**: Ускоряют поиск
**Минусы**: Замедляют добавление/изменение данных, занимают место

### Оптимизация баз данных
Зачем оптимизировать?
- Ускорить выполнение запросов
- Уменьшить нагрузку на сервер
- Улучшить отзывчивость приложения

#### Основные методы оптимизации:

1. **Правильные индексы**
- Создавать индексы на часто используемых полях
- Не создавать лишние индексы

2. **Эффективные запросы**
- Выбирать только нужные поля (не SELECT *)
- Избегать функций над индексируемыми полями
- Использовать LIMIT для ограничения результатов

3. **Нормализация базы данных**
- Устранить дублирование данных
- Разбить данные на логические таблицы

4. **Кэширование**
- Сохранять результаты частых запросов
- Использовать Redis или Memcached

5. **Партиционирование**
- Разделить большую таблицу на меньшие части
- Например, по датам (данные за каждый год в отдельной партиции)
